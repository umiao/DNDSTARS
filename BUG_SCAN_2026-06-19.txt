仍存在的主要 bug / 风险

[P0] 玩家端激活大多数主动特性仍然不是 DM 权威结算，可能导致 AP、状态、特性效果两端不一致
位置：
- src/pages/MapsPage.tsx:3476-3554
- src/pages/MapsPage.tsx:5514-5547
- src/lib/sharedApi.ts:71-84

现象：
- 玩家端激活特性时，只有 preciseStrike / 精准打击通过 player-action 发给 DM。
- doubleArrow / 双箭、eagleEye / 鹰眼、trackingArrow / 追踪箭等仍在玩家端本地直接 spendAP/updateChar/useClassFeature。
- 但 sharedApi 在玩家端且战斗进行中会拒绝写 characters/maps：
  if ((name === 'characters' || name === 'maps') && sharedCombatIsActive()) return
- 结果是：玩家端 UI 可能显示“已激活/已扣 AP”，但 DM 权威状态不知道；后续攻击由 DM 结算时，DM 可能看不到 doubleArrowReady、AP 扣除或其他 buff。

影响：
- 双箭可能在玩家端看起来已激活，但 DM 端攻击结算不吃双箭。
- 玩家端扣掉 AP，DM 端 AP 不扣，之后同步回来可能回弹。
- 类似问题会影响所有还没接入 player-action 的主动特性。

建议修法：
- 将所有主动特性统一走 player-action。
- 玩家端可以先做 optimistic UI，但必须发送 featureKey 给 DM。
- DM 验证 AP/次数/条件后更新权威角色状态，再 ack 给玩家端。
- handlePlayerActionRequest 目前只支持 preciseStrike，需要扩展为通用 activate-feature dispatcher。


[P1] 玩家攻击动作的语义去重窗口过粗，可能挡住合法的第二次同目标攻击
位置：
- src/pages/MapsPage.tsx:5443-5475
- src/pages/MapsPage.tsx:5508-5512

现象：
- 当前去重 key 包含 combatId、round、initiativeIndex、actorTokenId、characterId、type、skillId、targetTokenId/targetTokenIds、targetCell 等。
- 去重窗口为 8000ms。
- 这能挡住重复广播导致的双投掷，但也会误挡同一回合内合法的第二次攻击：
  例如角色有 2 AP，连续用基础射击攻击同一个目标，第二次可能被判为 duplicate-action。

影响：
- 解决了“同一动作重复处理”的同时，可能引入“合法连击/第二次普攻被拒绝”的新问题。

建议修法：
- 不要只用语义 key 去重。
- 更好的做法是：
  1. 玩家端每个动作带 clientActionId。
  2. DM 端保存“正在处理”和“已处理”的 actionId。
  3. 仅把同一个 clientActionId 的重复投递视为重复。
  4. 对 click/tap 双触发可在 UI 层锁 pendingPlayerAction，而不是在 DM 层拦截所有同语义动作。
- 如果还要保留语义兜底，窗口应极短，例如 300-600ms，并且不覆盖基础射击这类可重复动作。

[P2] 旧骰子逐帧同步代码已经禁用，但仍保留大量旧路径，后续维护容易再次引入双渲染
位置：
- src/pages/MapsPage.tsx:92
- src/pages/MapsPage.tsx:1124-1210
- src/pages/MapsPage.tsx:6523-6573

现状：
- ENABLE_LEGACY_DICE_STREAM = false。
- publishDiceStream 当前为 no-op。
- 旧 sharedDicePreview / pendingDiceStreamsRef / dice-stream 订阅和渲染逻辑仍保留，只是被常量开关挡住。

影响：
- 目前测试通过，但代码里同时存在两套骰子同步模型：
  1. 旧 dice-stream / frame replay。
  2. 新 dice-roll-request / 固定结果回放。
- 后续改骰子时很容易误开旧路径，导致再次出现“一次结算两颗骰子/旧队列骰子飞出来”。

建议修法：
- 如果确定不用逐帧回放，删除旧 dice-stream 类型、订阅、sharedDicePreview 渲染、pendingDiceStreamsRef。
- 如果要保留，至少写测试确保 ENABLE_LEGACY_DICE_STREAM=false 时不会响应任何 dice-stream 事件。


[P2] 骰子组件存在随机 fallback，极端情况下仍可能出现动画值与结算值不一致
位置：
- src/components/DiceBoxRollOverlay.tsx:30-32, 97-100, 150
- src/components/DiceBoxD20Overlay.tsx:17-20, 106-109

现象：
- 正常路径中 MapsPage 会先生成 values/value，再传给 overlay，所以两端应一致。
- 但如果某个调用没有传 values，或 iframe timeout 且 value 缺失，组件会 fallback 到 Math.random。

影响：
- 大多数当前战斗路径不触发该问题。
- 但如果未来新增骰子调用忘记传固定值，就可能复现“动画点数、log、实际结算不一致”。

建议修法：
- DiceBoxRollOverlay / DiceBoxD20Overlay 不应自行随机生成权威结果。
- 若缺少 forced value，应报错或显示“骰子失败”，由调用方/DM 端重新发起权威骰值。
- 至少在 dev 环境 console.error 提醒。


[P2] 端到端测试缺少“同一回合连续两次同目标基础攻击”的覆盖
位置：
- e2e/combat-regression.spec.ts
- e2e/monster-ap.spec.ts

现状：
- 已有 e2e 覆盖很多关键流程，但没有覆盖：
  - 玩家 2 AP 连续攻击同一目标两次。
  - 玩家端激活双箭后，DM 端实际吃到 doubleArrowReady。
  - 玩家端激活非精准打击的主动特性后，DM 端权威 AP/次数同步。

影响：
- 当前 [P0] 和 [P1] 这类问题，现有 e2e 不一定能抓到。

建议补测：
- 新增 e2e：
  1. 玩家回合 2 AP。
  2. 对同一敌人使用基础射击两次。
  3. 断言两次都被 DM accepted，AP 从 2 到 0，log 有两次攻击，骰子/伤害结算两次。
- 新增 e2e：
  1. 玩家端激活双箭。
  2. DM 端确认 AP 扣除、doubleArrowReady 为 true。
  3. 基础射击时 DM 端结算双箭额外骰和次数消耗。


[P3] 骰子 overlay 仍有大量 console.info/debug 输出，长时间游玩会污染控制台
位置：
- src/components/DiceBoxRollOverlay.tsx:80-90, 128-130
- src/components/DiceBoxD20Overlay.tsx:78-86, 119-121

影响：
- 不影响结算。
- 但调试长战斗、自动化测试或用户反馈日志时会噪音很大。

建议修法：
- 加 dev-only 开关，例如 import.meta.env.DEV && DICE_DEBUG。
- 默认关闭。


结论
----
- 当前编译、单元测试、e2e 全部通过。
- 但 repo 不是完全健康状态：
  1. lint 明确失败。
  2. 玩家端主动特性仍有 DM 权威同步漏洞。
  3. 新增的攻击去重有误杀合法第二次攻击的风险。
  4. 骰子同步旧路径仍残留，维护风险较高。

建议下一步优先级
----------------
1. 先修 [P0]：把所有主动特性接入 DM 权威 player-action。
2. 再修 [P1]：替换当前 8 秒语义去重，改为 actionId/transactionId 级别去重。
3. 补 e2e：连续同目标攻击、双箭玩家端激活到 DM 端结算。
4. 清理旧 dice-stream 死代码。
5. 最后处理 lint，决定是修代码还是调整 eslint 规则。
