# 战斗同步修复工作流

目标：把战斗状态改成 DM 端权威结算，玩家端只提交“意图”和本地交互确认。每修复一个问题单独提交，最终补齐单元测试并再次扫描。

## 权威原则

1. DM 端是战斗状态唯一写入者：生命值、AP、回合、先攻、怪物行动、伤害结算都由 DM 端计算并广播。
2. 玩家端只负责提交动作请求：结束回合、移动、攻击、是否闪避、是否发动反应等。
3. 所有跨端消息必须带自增 `id` 或稳定 `requestId`，接收端只处理比已处理消息新的请求。
4. 玩家端发起动作后进入锁定状态，直到 DM 端返回结果并同步完状态，再延迟 100ms 解锁。
5. 闪避、借机、特性插入都作为 DM 队列里的小事务执行，不能由玩家端直接改血量或 AP。

## 修复清单

### 1. DM 战斗状态唯一写入

- `combat`、战斗中的 `maps`/`characters` 权威字段只允许 DM 端写入。
- 玩家端不能本地推进先攻、回合或 AP。
- 验收：玩家端刷新或动作不会覆盖 DM 端血量、AP、token 位置。

### 2. 玩家动作请求与 DM 确认

- 玩家端的结束回合、移动、攻击、激活特性只发请求。
- DM 端消费请求、扣 AP、推进回合、广播结果。
- 验收：未轮到玩家时玩家端操作禁用；结束回合不会卡在等待 DM 结算。

### 3. 闪避流程

- 怪物攻击玩家时，DM 端扣怪物攻击 AP 并创建闪避请求。
- 玩家端只确认是否闪避；若选择闪避，DM 端扣玩家 AP 并使用玩家端骰值/请求结果进行判定。
- 15 秒未响应自动视为承受伤害。
- 验收：闪避成功不掉血；闪避失败后由 DM 端伤害结算并广播。

### 4. 骰子广播统一

- 只保留一条权威骰子事件路径。
- 骰值先确定，再由两端播放同一次结果动画。
- 验收：DM/玩家端骰面、Log、实际结算值一致。

### 5. AP 与回合

- 每个新战斗轮开始时，DM 端统一恢复所有单位 AP。
- 怪物攻击和移动由 DM 端扣 AP，连续行动必须等待上一次结算结束。
- 验收：怪物二次攻击扣 AP；玩家新轮次 AP 正确恢复；同一轮中已用掉的闪避 AP 不在玩家回合开始时回满。

### 6. 单元测试

新增测试覆盖：

- 开始战斗与 AP 初始化。
- 激活特性扣 1 AP。
- 移动扣 1 AP。
- 攻击扣 1 AP。
- 闪避成功：扣闪避 AP，不造成伤害。
- 闪避失败：扣闪避 AP，进入伤害判定并扣血。
- 玩家端请求不直接改变权威血量/AP。

### 7. 最终扫描

- 运行 `npm test`。
- 运行 `npm run build`。
- 搜索残留的玩家端权威写入、重复骰子通道、无序共享写入。
- 生成中文修复报告。
